## 1. Executive Summary

ExpenseFlow 시스템에 대한 Playwright 자동화 QA 감사를 실시한 결과, 총 **7건의 이슈**가 발견되었다.

- CRITICAL 1건: 비용 제출 후 데이터가 목록에 표시되지 않음 (DB 저장 또는 인증 연계 문제)
- HIGH 4건: 법카사용 폼의 기능 미구현 3건 + 비용 상세 테스트 불가 1건
- MEDIUM 2건: 법카사용 폼 긴급 체크박스 누락, 입금요청 제출 후 확인 UX 미흡

입금요청 폼은 핵심 기능(VAT 계산, 원천징수, 선지급, 카테고리 토글, 은행 검색 등)이 안정적으로 동작하며, 데스크톱/모바일 양쪽 뷰포트에서 페이지 렌더링이 정상적으로 확인되었다. 다만 BYPASS_AUTH 모드에서 API 인증 우회가 불완전하여 일부 API 호출이 401 에러를 반환하고 있으며, 이것이 CRITICAL 이슈의 근본 원인으로 판단된다.

**종합 판정**: 핵심 데이터 흐름(제출 -> 저장 -> 조회)이 정상 동작하지 않아, 현재 상태로는 production 배포가 불가하다. CRITICAL 이슈 해결 후 재감사가 필요하다.

## 3. 발견된 이슈 목록

### CRITICAL (1건)

#### ISS-001: 비용 제출 후 목록에 데이터 미표시

| 항목 | 내용 |
|------|------|
| **심각도** | CRITICAL |
| **영역** | 비용 목록 (Expense List) |
| **재현 단계** | 입금요청 또는 법카사용 폼에서 정상 제출 -> 비용 목록 페이지로 이동 -> 목록이 비어있음 |
| **예상 동작** | 제출한 비용 건이 목록에 표시되어야 함 |
| **스크린샷** | `e2e-screenshots/15_list_after.png`, `e2e-screenshots/11_expense_list_after.png` |

**원인 분석**:

BYPASS_AUTH 모드에서 페이지 렌더링(SSR/CSR)은 인증을 우회하지만, API route(`/api/expenses`)에서는 인증 미들웨어가 여전히 동작하여 401 Unauthorized를 반환하고 있다. 이로 인해 두 가지 시나리오가 가능하다:

1. **제출 시 API 호출 실패**: 폼 제출 API가 인증 실패하여 DB에 데이터가 저장되지 않음
2. **조회 시 API 호출 실패**: 데이터는 저장되었으나, 목록 조회 API가 인증 실패하여 빈 목록 반환

`report.json`에서 `/api/expenses?limit=1`이 401을 반환한 것으로 보아, 최소한 조회 API에서 인증 문제가 확인된다. 제출 API도 동일한 경로(`/api/expenses` POST)를 사용할 가능성이 높으므로, 두 시나리오 모두 해당될 수 있다.

**우선순위**: P0 - 즉시 수정 필요

## 5. 네트워크/인증 관련 참고사항

### 401 Unauthorized 에러 (4건)

| # | 실패한 API Endpoint | HTTP Status |
|---|---------------------|-------------|
| 1 | `/api/admin/dashboard?period=this_month` | 401 |
| 2 | `/api/expenses?type=DEPOSIT_REQUEST&status=SUBMITTED&limit=100` | 401 |
| 3 | `/api/notifications` | 401 |
| 4 | `/api/expenses?limit=1` | 401 |

### 분석

`BYPASS_AUTH=true` 환경변수는 현재 **페이지 레벨 렌더링**에서만 인증을 우회하고 있다. API route handler(`src/app/api/` 하위)에서는 Supabase 인증 미들웨어가 여전히 활성화되어 있어, 클라이언트 측 API 호출이 실패한다.

이는 다음과 같은 구조적 문제를 시사한다:

1. **인증 우회의 불완전한 적용**: 페이지 SSR과 API route에서 인증 체크 로직이 분리되어 있으며, BYPASS_AUTH 플래그가 API route까지 전파되지 않음
2. **Mock 사용자 ID 불일치**: BYPASS_AUTH 모드에서 사용하는 mock user ID가 실제 DB의 users 테이블에 존재하지 않을 가능성
3. **Supabase RLS(Row Level Security)**: DB 레벨에서 RLS 정책이 활성화되어 있다면, 유효한 Supabase 세션 없이는 데이터 CRUD가 불가능

### 영향 범위

관리자 페이지(대시보드, 승인 대기, 리포트, 사용자 관리)는 렌더링은 되지만, API 호출 실패로 인해 실제 데이터가 로드되지 않을 가능성이 높다. 이 영역은 본 감사에서 기능 검증이 아닌 렌더링 확인만 수행되었다.
